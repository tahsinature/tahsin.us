A few years ago, I ran an experiment to answer a simple question: **how much faster is gRPC than REST, really?** I built two Node.js microservices, wired them up with both protocols, and let the numbers speak. The results changed how I think about inter-service communication.

---

## The Setup

The experiment was straightforward. Two microservices — a **Master** service acting as a gateway, and a **Note** service acting as an internal backend. The Master service exposes both an HTTP/REST endpoint and a gRPC endpoint. Both ultimately do the same thing: make a call and return a response.

## Why gRPC?

Before diving into the numbers, it's worth understanding what makes gRPC fundamentally different from REST.

REST typically runs over **HTTP/1.1**, uses **JSON** for serialization, and establishes a new TCP connection (or reuses one with keep-alive) for each request. Every request carries full HTTP headers, and JSON must be parsed from text on both ends.

gRPC runs over **HTTP/2**, uses **Protocol Buffers** (protobuf) for serialization, and multiplexes multiple requests over a single persistent connection. The payload is binary — compact and fast to serialize/deserialize.

## Defining the Contract

With gRPC, you define your service contract using Protocol Buffers. Here's what the proto definitions looked like:

```protobuf
// master.proto
syntax = "proto3";
package Master;

service AllService {
  rpc GetTest (Empty) returns (Success);
}

message Empty {}

message Success {
  bool success = 1;
}
```

```protobuf
// note.proto
syntax = "proto3";
package Note;

service MasterService {
  rpc GetUserNote (UserId) returns (Notes);
}

message UserId {
  string id = 1;
}

message Note {
  string title = 1;
  string body = 2;
}

message Notes {
  repeated Note notes = 1;
}
```

The proto file is the single source of truth. Both the server and client are generated from it — no more mismatched request/response shapes, no more guessing what an endpoint returns.

## The Benchmark

The benchmark script fires sequential calls in a loop, timing the total duration for N requests. Simple, but effective for measuring per-call latency accumulation.

```typescript
// Simplified benchmark logic
const start = performance.now();

for (let i = 0; i < numberOfCalls; i++) {
  if (protocol === "grpc") {
    await grpcClient.GetTest({});
  } else {
    await axios.get("http://localhost:3000/health");
  }
}

const duration = performance.now() - start;
console.log(`${numberOfCalls} calls took ${duration}ms`);
```

## The Results

I ran the benchmark on a MacBook Pro (Intel i7-8750H, 16GB RAM). Here's what I found:

| Calls | REST | gRPC | Speedup |
|------:|-----:|-----:|--------:|
| 10 | 46.8ms | 8.9ms | **5.3x** |
| 100 | 248.7ms | 38.9ms | **6.4x** |
| 1,000 | 1.75s | 228.8ms | **7.6x** |
| 10,000 | 17.04s | 2.91s | **5.8x** |
| 50,000 | 187.04s | 15.18s | **12.3x** |

Read that last row again. **50,000 calls: REST took over 3 minutes. gRPC took 15 seconds.** That's a 12.3x difference.

## Why the Gap Widens

The most interesting observation isn't just that gRPC is faster — it's that the **advantage grows with volume**. At 10 calls, gRPC is ~5x faster. At 50,000 calls, it's over 12x faster.

This makes sense when you consider what's happening under the hood:

1. **Connection reuse**: gRPC's HTTP/2 connection is established once and multiplexed. REST pays connection overhead repeatedly.
2. **Binary serialization**: Protobuf encoding/decoding is significantly cheaper than JSON parsing, and the savings compound over thousands of iterations.
3. **Header compression**: HTTP/2's HPACK compression means headers shrink after the first request. HTTP/1.1 sends full headers every time.

At low volumes, the per-call overhead is small in absolute terms. But multiply it by tens of thousands, and the waste becomes enormous.

## When to Actually Use gRPC

After running this experiment, it might be tempting to slap gRPC on everything. Don't. Protocol choice is a trade-off, and context matters.

**gRPC shines when:**
- You have **service-to-service communication** in a microservices architecture
- You need **high-throughput, low-latency** internal APIs
- You want **strongly typed contracts** enforced at the protocol level
- You're building **streaming APIs** (gRPC supports bidirectional streaming natively)

**REST is still the better choice when:**
- You're building a **public-facing API** that needs to be easily consumed by any HTTP client
- Your consumers are **browsers** (gRPC-Web exists but adds complexity)
- **Human readability** of payloads matters for debugging
- Your team doesn't need the performance characteristics gRPC provides

In my experience at HaloJasa, introducing gRPC for inter-service communication tripled our API performance — closely mirroring the results from this experiment. The real world validated the benchmark.

## The Modern Landscape

Since I ran this experiment, the gRPC ecosystem has matured significantly:

- **gRPC-Web** has stabilized, making it viable for browser clients with an Envoy proxy
- **Connect** (from Buf) offers a gRPC-compatible protocol that works natively in browsers without a proxy
- **Protocol Buffers v3** syntax is now the standard, with better defaults and simpler semantics
- Major cloud providers (AWS, GCP, Azure) all offer first-class gRPC support in their service meshes and load balancers
- **gRPC health checking** and **reflection** are now standardized patterns for production deployments

If you haven't revisited gRPC recently, the developer experience has improved dramatically.
